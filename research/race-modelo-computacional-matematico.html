01 · O que o RACE fez que ninguém fez

A inovação real da metodologia RACE

Frameworks de agentes existentes definem papéis — "esse agente pesquisa, esse escreve". O RACE faz algo fundamentalmente diferente: descreve como o agente pensa, usando instrumentos que a psicologia leva décadas refinando como parâmetros computacionais reais.

Perfil psicométrico como parâmetro de geração Inédito

DISC e Eneagrama não são texto no prompt — são pesos que determinam como o agente pondera velocidade vs. precisão, confronto vs. harmonia. Ancoragem: McClelland (1973) sobre competências mensuráveis, re-aplicado à IA cognitiva.

Níveis de skill como contrato de confiança Inédito

As estrelas 1–5 não são auto-avaliação decorativa — são o teto de autonomia que o sistema deposita em cada camada. Um agente com SQL 3/5 recebe revisão; com 5/5, executa. É um contrato operacional formalizado como tabela.

Dimensões soft como variável de design Inédito

Resiliência, colaboração e investigação como atributos computacionais de um agente. A literatura de HRI (Human-Robot Interaction) ainda está tentando formalizar isso. O RACE já usa.

02 · Arquitetura visual

Pipeline das 7 Camadas — Wireframe

Cada camada recebe um payload validado por Zod, processa, e emite um novo payload para a próxima. Nenhuma camada emite dados não-validados — fail-closed por design.

INPUT BRUTO · texto, modalidade: text
▼
CAMADA 1

PERCEPTION — Percepção

Normaliza o input bruto. Tokeniza. Classifica o taskType por keywords (migration, review, architecture, diagnosis, general). Única camada que aceita dado externo não-validado.

→ { normalizedText, tokens: ℕ, taskType: T }
▼
CAMADA 2

MEMORY — Memória de Longo Prazo

Recall episódico no Neon (tenant isolado por agente). Timeout 500ms fail-closed — se a memória não responde, o ciclo continua com recall vazio. Nunca bloqueia.

→ { tenantId, query, items[]: RecallItem[] }
▼
CAMADA 3

ATTENTION — Atenção / Working Memory

Monta o foco de trabalho priorizando os recall items mais relevantes + módulos/arquivos mencionados no input. Consome a memória — não a dispara (Ordem Quiron).

→ { focus, selectedItemIds[], mentionedModules[] }
▼
CAMADA 4

REASONING — Raciocínio

Chain-of-thought sobre scratchpad isolado. Extrai evidenceRefs (paths de arquivos referenciados). Deliberação estruturada — nunca sobre o workspace global.

→ { thesis, supportingItemIds[], evidenceRefs[] }
▼
CAMADA 5

PLANNER — Planejamento

Decompõe a tese em passos concretos e executáveis. Cada passo é prefixado com a evidência que o sustenta. Mínimo 1 passo — fail-closed se vazio.

→ { steps[]: string[] }
▼
CAMADA 6

ACTION — Ação

Realiza o draft da resposta a partir dos passos planejados. Marca evidenceCited como true se o draft contém marcadores de evidência de leitura real.

→ { draft: string, evidenceCited: boolean }
▼
CAMADA 7

EVALUATION — Gate de Domínio (imutável em runtime)

Afrodite: gate clínico — rejeita diagnóstico/prescrição/psicoterapia.
Atena: gate arquitetural — rejeita afirmação vaga sem evidência de código lido.
JARVIS OVERRIDE (root autenticado) suspende o gate por comando explícito.

→ { approved: boolean, score: [0,1], rejectionReason: string|null }
▼
CAMADA 8

REFLECTION — Reflexão (terminal)

Metacognição. Registra o resultado do ciclo. Confidence cai se houve rejeição na camada 7. Não retroalimenta a Percepção — sem loop no ciclo principal.

→ { note: string, confidence: [0,1] }
▼
OUTPUT · aprovado ou vetado com motivo auditável
03 · Formalização matemática

Modelo Matemático — Teoria dos Conjuntos

A formalização em conjuntos responde uma pergunta que a descrição não responde: o que exatamente entra e sai de cada camada? Sem isso, fronteiras ficam abertas à interpretação de quem implementa.

Universo e definições base

// Universo de dados do sistema 𝒰 = String × ℕ × Boolean × ℝ[0,1] × Set(String) // Conjunto de taskTypes possíveis 𝒯 = { architecture, review, migration, diagnosis, general } // Espaço de payloads de percepção E₁ = { x ∈ String | |x| > 0 } // input não-vazio S₁ = { (t, n, τ) | t ∈ String, n ∈ ℕ⁺, τ ∈ 𝒯 } // Função de transição da camada 1 f₁ : E₁ → S₁ f₁(x) = ( normalize(x), |tokens(x)|, classify(x) ) // Invariante da camada 1 — deve ser verdadeiro para todo s ∈ S₁ ∀ s ∈ S₁ : s.tokens > 0 ∧ s.taskType ∈ 𝒯 // Erro fail-closed: se modalidade ≠ text → lança UnsupportedModalityError

Cadeia completa de transições

// Cada camada Lᵢ é uma função fᵢ : Eᵢ → Sᵢ // A saída de Lᵢ é a entrada de Lᵢ₊₁: Sᵢ = Eᵢ₊₁ f₁ : E₁ → S₁ // PERCEPTION f₂ : S₁ → S₂ // MEMORY · S₂ = { tenantId, query, items[] } f₃ : S₂ → S₃ // ATTENTION · S₃ = { focus, selectedIds, modules } f₄ : S₃ → S₄ // REASONING · S₄ = { thesis, itemIds, evidenceRefs } f₅ : S₄ → S₅ // PLANNER · S₅ = { steps: List(String) } f₆ : S₅ → S₆ // ACTION · S₆ = { draft, evidenceCited } f₇ : S₆ → S₇ // EVALUATION · S₇ = { approved, score, rejectionReason } f₈ : S₇ → S₈ // REFLECTION · S₈ = { note, confidence } // Composição: o ciclo completo é a função composta Φ = f₈ ∘ f₇ ∘ f₆ ∘ f₅ ∘ f₄ ∘ f₃ ∘ f₂ ∘ f₁ Φ : E₁ → S₈ // Propriedade de composição: cada transição é bloqueante // ∀ i: fᵢ₊₁ só inicia quando fᵢ emite S com validação Zod OK

Gate de domínio — formalização

// Conjunto de marcadores de veto (Atena — gate arquitetural) V = { "acho que", "provavelmente", "talvez", "me parece", "suponho", "deve ser" } // Conjunto de marcadores de evidência (habilitam proposta) Ε = { "lido via filesystem", "evidência:", "grep mostra:", "arquivo lido:", "read [" } // Função de detecção vague(d) = ∃ v ∈ V : v ∈ normalize(d) cited(d) = ∃ e ∈ Ε : e ∈ normalize(d) // Regra de decisão da camada 7 f₇(d) = REJECT se vague(d) ∧ ¬cited(d) // vago SEM evidência APPROVE se ¬vague(d) // sem marcador vago APPROVE se vague(d) ∧ cited(d) // vago MAS com evidência OVERRIDE se root ∧ JARVIS_TRIGGER ∈ d // root suspende o gate // Score de aprovação score : S₇ → ℝ[0,1] score(s) = 1.0 se s.approved score(s) = 0.0 se ¬s.approved

Isolamento de tenant — garantia formal

// Cada agente opera sobre um tenant isolado no Neon 𝒜 = { afrodite, atena, hermes, musa, ... } // conjunto de agentes // Função de tenant tenant : 𝒜 → String (injetiva — agentes distintos → tenants distintos) // Garantia de isolamento ∀ a₁, a₂ ∈ 𝒜 : a₁ ≠ a₂ ⟹ tenant(a₁) ≠ tenant(a₂) ∀ recall ∈ S₂ : recall.tenantId = tenant(agente_atual) // cross-tenant falha
04 · Especificação computacional por camada

Ficha Computacional — Cada Camada

Para cada camada: domínio de entrada, função de transformação, imagem de saída, invariante verificável e comportamento de erro. Isso é o que transforma descrição em especificação.

1
PERCEPTION — Percepção Multimodal
EntradaE₁ = { x: String | |x| > 0, modalidade: "text" }
SaídaS₁ = { normalizedText: String, tokens: ℕ⁺, taskType: T }
Funçãof₁(x) = { normalize(x), count_tokens(x), classify_task(x) }
Invariantetokens > 0 ∧ taskType ∈ { architecture, review, migration, diagnosis, general }
ErroUnsupportedModalityError se modalidade ≠ "text" (fail-closed)
TempoSíncrono · O(n) onde n = |tokens|
2
MEMORY — Recuperação de Memória de Longo Prazo
EntradaS₁ (output da percepção)
SaídaS₂ = { tenantId: literal(agente), query: String, items: RecallItem[] }
Funçãof₂(p) = recall_neon(tenant, p.normalizedText, timeout=500ms)
InvariantetenantId = tenant_do_agente · items pode ser [] (fail-closed em timeout)
ErroTimeout → items = [] · NÃO lança exceção · ciclo continua
TempoAssíncrono · max 500ms · race com timeout
3
ATTENTION — Working Memory / Foco Atencional
EntradaS₁ × S₂ (percepção + memória)
Nota: MEMORY precede ATTENTION — ATTEND consome recall, não o dispara (Ordem Quiron)
SaídaS₃ = { focus: String, selectedItemIds: String[], mentionedModules: String[] }
Funçãof₃(m, p) = { top_relevance(m.items, p), filter_relevant(m.items, θ), extract_paths(p) }
Invariantefocus ≠ "" · θ = limiar de relevância configurável por agente
ErroSe m.items = ∅ → focus = p.normalizedText (fallback para percepção direta)
4
REASONING — Raciocínio / Chain-of-Thought
EntradaS₃ (output da atenção)
SaídaS₄ = { thesis: String, supportingItemIds: String[], evidenceRefs: String[] }
Funçãof₄(a) = { derive_thesis(a.focus), filter_supporting(a.selectedIds), extract_paths(a.focus) }
Invariantethesis ≠ "" · scratchpad ISOLADO (não toca workspace global)
DiferencialevidenceRefs = strings que parecem paths (contêm "/" ou ".ts" ou ".md")
5
PLANNER — Decomposição em Passos
EntradaS₄ (output do raciocínio)
SaídaS₅ = { steps: String[] } · |steps| ≥ 1
Funçãof₅(r) = decompose(r.thesis, r.evidenceRefs) → lista de ações concretas
Invariante|steps| ≥ 1 · fail-closed se steps = [] (erro de schema Zod)
6
ACTION — Realização / Draft
EntradaS₅ (output do planejador)
SaídaS₆ = { draft: String, evidenceCited: Boolean }
Funçãof₆(pl) = { realize(pl.steps), contains_evidence_marker(realize(pl.steps)) }
Invariantedraft ≠ "" · evidenceCited = cited(draft) conforme definição matemática
7
EVALUATION — Gate de Domínio (imutável em runtime)
EntradaS₆ (draft + evidenceCited)
SaídaS₇ = { approved: Boolean, score: ℝ[0,1], rejectionReason: String|null }
Funçãof₇(a) = gate_rule(a.draft, origin) — ver formalização matemática acima
Invariante¬approved ⟹ rejectionReason ≠ null (veto sempre auditável)
KernelIMUTÁVEL em runtime — só Rafael (sessão autenticada) altera o conjunto V de marcadores
Overrideroot ∧ "JARVIS OVERRIDE" ∈ input → gate suspenso para o comando atual
8
REFLECTION — Metacognição (terminal)
EntradaS₇ (verdict da avaliação)
SaídaS₈ = { note: String, confidence: ℝ[0,1] } ← output final do ciclo
Funçãof₈(e) = { metacognitive_note(e), e.score } · confidence cai se ¬approved
InvarianteNÃO retroalimenta PERCEPTION — sem loop no ciclo principal (DAG)
05 · Representação formal em grafo

Grafo G = (V, E) — Arquitetura das Camadas

Definição formal

// O ciclo cognitivo é um DAG (Directed Acyclic Graph) V = { P, M, At, R, Pl, Ac, Ev, Rf } // P=Perception, M=Memory, At=Attention, R=Reasoning // Pl=Planner, Ac=Action, Ev=Evaluation, Rf=Reflection E = { (P,M), (M,At), (At,R), (R,Pl), (Pl,Ac), (Ac,Ev), (Ev,Rf) } // Propriedades do grafo |V| = 8 // 8 nós |E| = 7 // 7 arestas directed // G é um DAG: não existe ciclo // ∄ caminho (v₁→v₂→...→v₁) — sem loop no ciclo principal // Toda aresta é BLOQUEANTE (síncrona): fᵢ₊₁ aguarda fᵢ // Exceção: aresta (P→M) tem timeout assíncrono de 500ms // Grau de entrada e saída de cada nó in(P)=0 out(P)=1 // fonte do DAG in(Rf)=1 out(Rf)=0 // sink do DAG ∀ nó intermediário: in=1, out=1 // pipeline linear

07 · Ancoragem científica

Literatura e Referências

O que ancora cada elemento do RACE na ciência estabelecida.

Formalização matemática de arquiteturas cognitivas

Anderson et al. (2004) — An Integrated Theory of the Mind
Psychological Review, 111(4), 1036–1060 · PMID: 15482072
Define ACT-R com equações formais de ativação e tempo de resposta por módulo. A lição histórica: a teoria de Anderson era boa desde os anos 70 — só virou ciência verificável quando ganhou formalização matemática. Ancora: a necessidade de modelo computacional além do teórico.
Dehaene, Lau & Kouider (2017) — What is consciousness, and could machines have it?
Science, 358(6362), 486–492 · PMID: 29074769
Global Workspace Theory formalizada como grafo de conectividade com transições de estado. Mostra que cognição em camadas com broadcast global (nossa Camada 3 — Atenção) tem base neurológica verificável. Ancora: pipeline de 7 camadas e o papel central da atenção.
Laird (2012) — The Soar Cognitive Architecture
MIT Press · ISBN: 9780262122962
Arquitetura em camadas (decisão/memória/aprendizagem) com gramática de produções e chunking formal. Demonstra como memória episódica e semântica operam em tenants separados. Ancora: isolamento de tenant por agente e memória de longo prazo.

Global Workspace Theory computacional

Baars & Franklin (2009) — Consciousness is computational: The LIDA model
Neural Networks, 20(9), 1198–1209 · PMID: 17998071 · DOI: 10.1016/j.neunet.2007.09.013
GWT implementada como ciclos cognitivos com grafos de ativação. Cada camada tem conjunto de entrada definido, função de transformação e saída. É a arquitetura mais próxima do nosso pipeline. Ancora: estrutura do ciclo cognitivo e broadcast da camada de Atenção.

Grafos em neurociência e cognição

Sporns (2011) — Networks of the Brain
MIT Press · ISBN: 9780262014984
Teoria de grafos aplicada a conectomas: modularidade, hubs, hierarquia. DAGs como representação de fluxo cognitivo sem ciclos. Ancora: representação formal G=(V,E) do pipeline e propriedade de acyclicidade.
Schmidhuber (2015) — Deep learning in neural networks: An overview
Neural Networks, 61, 85–117 · PMID: 25462637
Hierarquia de representação em camadas: cada camada aprende funções progressivamente mais abstratas sobre a saída da anterior. Formalização de composição funcional fᵢ₊₁ ∘ fᵢ. Ancora: composição das funções de transição e a cadeia Φ = f₈ ∘ ... ∘ f₁.

Psicometria como parâmetro computacional

McClelland (1973) — Testing for Competence Rather Than Intelligence
American Psychologist, 28(1), 1–14 · DOI: 10.1037/h0034092
Define competências mensuráveis como preditoras de desempenho — base científica para usar DISC e Eneagrama como parâmetros operacionais, não decorativos. Ancora: o uso dos perfis psicométricos do RACE como contratos de geração.
Marston (1928) — Emotions of Normal People
Kegan Paul · (base teórica do modelo DISC)
DISC originou-se de pesquisa clínica sobre comportamento em relação ao ambiente. Dominância, Influência, Estabilidade e Consciência como dimensões independentes de resposta comportamental. Ancora: validade científica dos eixos DISC usados pelo RACE.

Gate de domínio e safety em IA

Amodei et al. (2016) — Concrete Problems in AI Safety
arXiv:1606.06565 · DOI: 10.48550/arXiv.1606.06565
Define o problema de specification gaming — agente otimiza proxy errado sem restrição de domínio. O gate imutável da camada 7 é a resposta arquitetural direta a esse problema. Ancora: necessidade e design do gate de domínio por agente.

Para cada agente que vai receber 7 camadas, o RACE precisa entregar

1. Ficha computacional

Entrada · função · saída · invariante · erro para cada uma das 8 camadas. Sem isso o desenvolvedor infere.

O que o agente não pode emitir como output. Marcadores de veto, marcadores de aprovação, regra de override root.

3. Tenant de memória

Qual o tenantId isolado desse agente no Neon. Garante que recall não vaza entre agentes.

4. Wireframe do pipeline

O diagrama ASCII mostrando os payloads que cruzam cada transição. Serve para o artigo e para o desenvolvedor.
